# ease-design

A multi-runtime **design-cli**. Describe *what* you want; ease-design supplies all of
*how it looks*. A non-designer produces UI with the taste of a pro designer — by talking
to the agent CLI they already use (Claude Code, Codex CLI, Antigravity CLI).

> Distilled from [EaseUI](../EaseUI)'s design engine. Working name `ease-design`;
> the binary and slash-command namespace are `ui` / `/ui:*`.

## How it works

ease-design ships two runtime-neutral sources of truth and a thin per-runtime adapter:

- **`knowledge/`** — a plain-Markdown knowledge core (taste rubric, persona library,
  generation rules, component catalog, color science) the host model reads directly.
- **`ui` binary** — a deterministic Node CLI for non-LLM work (autofix, layout
  validation, token compilation, color math, component registry, export).
- **Adapters** — `ui init` generates the config a given agent CLI needs to know *when*
  to read the knowledge core and *when* to shell out to `ui`. The host CLI's own model
  generates HTML — no API keys.

A hard-gate critique→refine loop scores every generation across 6 taste axes + 1
consistency axis, so quality is enforced, not suggested.

## Install

No npm registry package — clone and build:

```sh
git clone <repo-url> ease-design
cd ease-design
npm install
npm run build
npm link          # exposes the `ui` binary on your PATH
```

Requires Node.js >= 20.

## Usage

```sh
ui --help
ui init --runtime claude     # wire ease-design into the current project
```

In an agent CLI, the workflows surface as `/ui:generate`, `/ui:iterate`,
`/ui:extract`, `/ui:from-ref`, `/ui:from-url`, `/ui:figma`, `/ui:redesign`,
`/ui:refine`, `/ui:slides` (Claude Code namespace).

`/ui:from-url <url>` extracts the live design system into `./<slug>/`
— a self-contained folder with `DESIGN.md` (Google-Labs alpha spec),
`DESIGN.preview.html` (open in a browser), the raw `source.html` +
`source.css` audit trail, `tokens.json` (frequency-ranked source
tokens with provenance), `run-summary.md`, and an `audit.md` report
whose exit code gates workflow success (5 audit families: format,
source-fidelity, ref-integrity, accessibility, discipline).

Flags: `--name <slug>` overrides the URL-derived folder name;
`--out-dir <path>` relocates the folder; `--force` overwrites an
existing folder without confirming.

## The non-designer happy path

The whole project is built around one user — someone with no design vocabulary
who wants production-quality UI on demand:

1. **Describe the intent in plain words.** `/ui:generate landing page for a new gym`.
   No mood-board, no color picker, no font list.
2. **ease-design picks personas + compiles a design system.** The host model
   reads `knowledge/persona-index.md`, scores the intent against 23 curated
   personas, and selects a diverse top-K. Then `ui ds init` compiles a
   project-scoped DS — semantic tokens, component registry, hash-sealed
   manifest — from the chosen persona + intent.
3. **Three variants come back.** Each variant draws from a different selected
   persona, so the user picks by eye rather than by vocabulary. The 7-axis
   critique gate scores each one before it's shown; failing variants regenerate
   silently.
4. **Refine in plain words.** `/ui:iterate make the hero warmer, tighten the
   headline`. The host model translates that into token / layout edits; the
   critique gate scores the new version; the DS hash-seal stays intact.

No keys to manage, no design-system spec to write, no taste vocabulary to
learn. The host CLI is the only interface.

## Status

**v1 — feature-complete, end-to-end synthetic walkthrough green, live dogfood
pending.**

- `knowledge/` core: 7-axis taste rubric, 23 personas across 7 families, 32
  components across 8 categories, color science, token taxonomy, prompt modes.
- `ui` binary: 11 commands covering color math, token compilation, autofix,
  layout validation, component registry, edit strategy, JSON streaming,
  standalone export, manifest + adapter generation, design-system SSOT
  (`ds init / context / change-token / status`).
- Workflows: 9 host-model workflows (`generate`, `iterate`, `extract`,
  `from-ref`, `figma`, `redesign`, `refine`, `slides`, `critique`) + 6 skill
  wrappers, both per-runtime adapter-generated by `ui init`.
- Critique gate: hard pass/fail loop scoring every generation against all 7
  axes before the user ever sees it.
- 619 unit tests passing; zero runtime dependencies (devDeps only); all four
  CI gates green.

**What's still pending for a future session:**

- **Live dogfood.** A host AI model running `/ui:generate`, `/ui:iterate`,
  `/ui:extract` end-to-end against the installed binary. The synthetic
  walkthrough under `examples/synthetic/` covers the deterministic surface;
  live dogfood will populate `examples/generated/`, `examples/iterated/`, etc.
- **Threshold tuning of the taste rubric.** The 0-10 pass cutoffs per axis are
  current best-guesses. Real outputs from live dogfood let us calibrate so the
  gate catches bad designs without rejecting good ones.

## Repo layout

| Path | Purpose |
|---|---|
| `knowledge/` | Markdown knowledge core — taste rubric, personas, components, color science, token taxonomy, prompt modes |
| `src/cli.ts` | `ui` binary entrypoint + subcommand router |
| `src/commands/` | one file per `ui` subcommand |
| `src/core/` | shared deterministic logic — color math, tokens, registry, autofix, validation |
| `src/adapters/` | per-runtime adapter templates for `ui init` |
| `schemas/` | JSON Schemas — `design.tokens.json`, component registry, DS manifest |
| `templates/` | workflow + skill Markdown templates (source of truth for adapter generation) |
| `examples/` | synthetic walkthrough + (post-dogfood) generation outputs |
| `docs/` | audit-gate report, journals |
